> **导读**：用 AI 写代码时，最让人头疼的不是它写出 bug，而是它「改对了 A，却忘了改 B」。问十个工程师，九个遇到过。这篇文章不谈 prompt 技巧，而是回到一个更根本的问题——**那些被遗忘的关联，到底应该以什么形式存在于你的工程里**。文档？RAG？还是别的？答案是：大部分关联根本不该靠「记」，而该靠工具强制。

## 一、先看清根因：它不是「忘了」，是「从未看到」

很多人把「改 A 忘改 B」归咎于 AI 记性差、上下文不够长，于是本能地去找两个方向的解药：

- 让 AI 更聪明（更长的上下文、更强的模型）；
- 让 AI 记更多（写更多文档、做 RAG 把知识喂给它）。

**这两个方向都跑偏了。** 真正的根因是：

> 关联丢失，是因为这条关联没有被**显式化、机器可强制执行地**表达出来。

只要一条关联还停留在「需要 AI 自己去发现、去记住」的层面，它就一定会丢。原因不是 AI「忘了」——而是当上下文一长、文件一多，AI **从始至终就没"看到"** 那条关联。它没有在你的上下文窗口里出现过，它不在 AI 当前读的那几个文件里，自然就无从改起。

所以正确的方向正好反过来：

> **减少对 AI 记忆的依赖，增加工具的强制力。** 让工具（编译器、lint、grep、测试、hook）去保证关联不断，AI 只负责动手改，工具负责验证关联没断。

这是一个反直觉但关键的判断。后面所有的手段，都围绕这个判断展开。

## 二、关联有两种：代码能表达的，和代码表达不了的

在讨论「怎么记」之前，必须先分清一条关联属于哪一种。因为这决定了该用什么手段。

| 关联类型 | 例子 | 谁来保证 |
|---------|------|---------|
| **代码能表达** | 函数签名、import 依赖、类型引用、函数调用 | 编译器 / 类型系统 / lint / grep |
| **代码表达不了** | 「改了订单状态，要同步改三处分支逻辑」「前端表单字段名要和后端接口对齐」 | 只能靠记 / 生成 / 测试 |

代码能表达的关联，你基本不用操心——改了函数签名，调用方编译就报错；删了导出，lint 会告诉你哪里还在引用。这部分工具会兜住。

**真正让你头疼的，全是第二类：代码表达不了的隐性语义关联。** 它们在物理上、语法上是分开的——没有类型依赖、没有 import、没有直接调用——但在业务上、逻辑上、语义上是绑定的。改 A 必须同步改 B，而编译器和 grep 都抓不到。

下文要穷举的 13 类，全是这第二类。这是你真正的战场。

## 三、判断标准：一条信息到底该不该「记下来」

在展开 13 类之前，先给一个贯穿全文的判断标准。无论是关联，还是别的什么知识，该不该落到文档/记忆里，只看一个问题：

> **代码 / 类型 / 测试 / 工具，能不能自动回答它？**

- **能** → 绝不记。记了必然过时，反而误导下一次的 AI。
- **不能，且丢了它会导致 AI 或人反复出错、做出错误决策** → 必须记。

换句话说，文档该记的是**代码拒绝表达或表达不了的东西**。凡是代码已经能回答的（函数签名怎么写、目录结构长什么样、这个函数怎么工作），都不要记——那是给代码和自动生成文档抢「唯一真相」，必然过时。

这个标准会反复用到：它既是「关联该不该记」的标准，也是「除了关联，还有什么该记」的标准。

## 四、13 类「代码表达不了的隐性关联」穷举

这是全文的主体。下面每一类都是高频踩坑点，我给定义、给一个通用例子、解释为什么代码抓不到。你可以逐条对照自己的项目——「莫名其妙就丢」的关联，基本都能在这里对号入座。

### 1. 镜像冗余：同一概念的多个投影

一个概念在 N 处各定义了一遍，编译器看不出它们是「同一个东西」。

> **例子**：一个「用户」概念，在前端有个 TypeScript 的 `User` 类型，后端有个 `User` 结构体，数据库有张 `users` 表，OpenAPI 文档里有一份 schema，还有一份错误码枚举也引用了用户字段。五处各写一遍，互相之间没有任何编译期联系。

改了任意一处的字段定义，其余四处都是「副本」，必须手动同步。漏一个，就是 bug。**这是所有隐性关联里最高发的一类。**

### 2. 成对共变：A 出现，B 必须跟着出现

加了 A，就必须加 B，但没有任何机制强制。

> **例子**：新增一个 `/reports/export` 路由，就必须同时在导航菜单、权限配置表、面包屑映射里各登记一次。路由本身不会告诉菜单「我来了」。

漏登任何一处，表现就是「接口能用，但用户在界面上找不到入口」或者「有入口但点进去 403」。

### 3. UI ↔ 数据 ↔ 校验三角

前后端分离项目的重灾区。同一个字段，在三个地方各表达一次。

> **例子**：一个「手机号」表单字段——前端有一份校验正则，后端有一份校验逻辑，提交时 JSON body 里有个字段名。三处共变。改了正则规则或字段名，另两处不跟，要么校验不一致，要么提交直接报字段不存在。

### 4. 状态机 / 语义映射

一个状态值或枚举值，对应散落在多处的分支行为。

> **例子**：订单状态 `PAID`，要触发「发货流程 + 通知买家 + 财务记账」三件事，这三件事分别写在三个不同模块的 `switch (status)` 里。改了状态机（比如新增一个 `PARTIAL_PAID`），所有按状态分支的地方都要跟。

### 5. 跨实体不变量

两个独立实体之间必须成立的关系，局部代码看不出来。

> **例子**：账户的「可用余额 + 冻结金额 = 总余额」必须时刻成立。但这三个字段分散在不同表、不同服务里，没有哪一行代码能表达这个等式。

违反这个不变量，账就对不上——而这种 bug 往往要等到月底对账才被发现。

### 6. 因果事件链

A 的发生会触发 B、C、D，链路跨越模块甚至消息队列。

> **例子**：用户下单 → 扣库存 → 记财务流水 → 发通知 → 加积分。这条链可能跨过一个消息队列，每一步是独立的消费者。改了链路上的某一步（比如扣库存改成预扣），下游所有 handler 都可能受影响，但它们互相看不见彼此。

### 7. 生命周期配对：创建 ↔ 销毁

创建 A 的时候要建 B，销毁 A 的时候必须销毁 B。泄漏的根源。

> **例子**：前端组件里 `setInterval` 启动了一个定时器，或者 `websocket.subscribe()` 订阅了一个频道。组件卸载时必须 `clearInterval()` / `unsubscribe()`。漏了销毁，就是内存泄漏、重复请求、幽灵回调。

### 8. 时序 / 顺序关联

「A 必须在 B 之前执行」——代码看得出顺序，但看不出「为什么必须这个顺序」。

> **例子**：服务启动时，「先建立数据库连接池，再启动依赖数据库的定时任务」。顺序反了，定时任务第一轮就会因为拿不到连接而崩。数据库迁移脚本也同理：必须按版本号顺序执行，跳号或乱序会损坏数据。

### 9. 版本 / 兼容配对

必须配对，否则错位。

> **例子**：客户端 v2 的协议格式，必须配服务端 v2 的接口；数据库迁移脚本的版本号，必须和当前代码版本对应。只升级一头，就会出现「客户端发 v2 报文，服务端还在按 v1 解析」的错位。

### 10. 互斥 / 冲突关联

A 和 B 不能同时成立。

> **例子**：配置项「调试模式」和「生产模式」互斥——开了调试就不能上生产；业务上「同一用户不能同时持有两张有效主卡」。这种约束没有类型能表达，只能靠校验和测试守护。

### 11. 权限 / 可见性矩阵

同一个权限规则，散落在四处各实现一遍。

> **例子**：一个「删除」权限——路由守卫判断一次、API 中间件判断一次、UI 上按钮的显隐判断一次、数据查询时的过滤又判断一次。四处各写一遍同样的逻辑。改了权限规则，漏改任何一处，就是一个越权漏洞。

这是隐性关联里**安全代价最高**的一类。

### 12. 命名 / 术语映射

同一个业务概念，在不同层叫不同的名字。

> **例子**：业务方管它叫「工单」，代码里是 `Ticket`，API 字段是 `ticket`，数据库表是 `tickets`，而产品文档里又叫「事项」。这张映射表如果不显式记下来，AI 在不同层之间翻译时就会不一致——同一个东西，文档里叫「工单」，新写的 API 里突然变成了 `task`。

### 13. 测试 / 文档 ↔ 实现

改了实现的行为，测试和文档要跟；反过来，测试就是行为契约。

> **例子**：分页逻辑从「offset 分页」改成了「游标分页」，那么测试用例要改、API 文档要改、对外 SDK 的示例也要改。如果只改了实现，测试就会红，或者更糟——测试也跟着悄悄改对了，但文档还停在旧逻辑，下一个看文档的人就被带偏。

---

把这 13 类放一起看，你会发现一个共同点：**它们的共同敌人不是「复杂性」，而是「分散」**。一个真相散落在多处，每一处都正确，但它们之间的「必须一致」关系，没有任何一行代码能表达。这才是 AI（以及人类工程师）频繁漏改的根本结构原因。

## 五、除了关联意图，还有哪些必须记下来

关联意图只是「该记」的内容之一。按第三节的判断标准往下推，还有一大批信息是代码表达不了、但又必须持久化的。按优先级排列：

| 优先级 | 类目 | 为什么代码表达不了 |
|--------|------|-------------------|
| **极高** | **禁忌 / 反例 / 踩坑**（「别用 X，因为 Y」「试过这方案，挂在 Z 上」） | 代码只看得到「现在这么做」，看不到「曾经那么做炸了」。负向知识几乎只能靠记，且防的是灾难 |
| **极高** | **已被纠正过的错误认知** | 防止 AI 下一次原地复犯 |
| 高 | **决策理由 + 被否决的方案**（为什么选 A 不选 B） | 代码只有结果，没有「为什么不选那个」。否则 AI 会把「刻意为之」当 bug 给你「修」掉 |
| 高 | **不变量 / 跨代码约束** | 局部代码看不出全局不变式 |
| 中 | **外部 / 环境假设**（线上数据是脏的、某第三方 API 的非文档化行为） | 来自踩坑，代码里看不到「这个假设的来源」 |
| 中 | **业务 / 领域规则** | 代码实现的是某个 case，业务规则更广 |
| 中 | **工程约定 / 流程**（命名、目录、提交、重构必须先 grep） | meta 规则，代码本身不携带 |
| 低 | **项目状态 / 阶段性约束**（「下周交付，别动 X 模块」） | 不能从代码或 git 推导 |

**如果只能记一类，优先记「别这么做 + 为什么」。** 因为正向描述（这是什么、怎么用）代码和自动生成文档已经覆盖得最好；唯独「别这么干、试过不行、这样会炸」这种负向知识，代码完全无法表达，只能靠记，而它防的恰恰是最贵的事故。

反过来，也要知道**什么不该记**，避免文档膨胀腐化：

- 代码已经能回答的（结构、签名、调用关系）；
- 能自动生成的（API 文档、类型文档、依赖图）；
- 一次性的临时状态（「正在修的这个 bug 到哪一步了」）；
- 还没沉淀、会被快速推翻的临时决策。

原则一句话：**文档要记的是代码拒绝表达的东西，而不是重复代码已经表达的东西。**

## 六、可检测性光谱：把「隐性」推向「显式」

这是全文最该带走的一个思维模型。所有关联，按**能不能被自动检测**，排成一条光谱：

```
可编译期检测 ── 可静态分析 ── 可运行时检测 ── 完全隐性
(类型/import)    (grep/lint/AST)   (测试/契约)     (只在脑子里/文档里)
   ✅ 工具保证       ✅ 工具半保证       ✅ 测试保证       ❌ 痛点在这
```

第四节那 13 类关联，默认都在光谱最右端——「完全隐性」。而**你的目标，是把它们尽可能往左推**。推得越靠左，越不依赖 AI 或人的记忆，越不可能丢。

- 一条「前后端 schema 必须一致」的关联（第 1 类），如果靠文档记，它在最右端，必丢；
- 如果你写一个契约测试去断言「前端类型和后端 schema 一致」，它就被推到了「可运行时检测」，从此丢不了；
- 如果你更进一步，用一处 schema 定义生成前后端代码，它就被推到了「可编译期检测」，根本不可能不一致。

**推不动的那一点残余，才轮到文档去记。** 下一节讲怎么推。

## 七、把关联推成显式的分层手段

针对隐性关联，按强度从高到低，有六种手段。注意：**「写文档」是最弱的，应该作为最后手段**，而不是第一反应。

| 手段 | 强度 | 适用场景 |
|------|------|---------|
| **单一真相源（SSOT）+ 代码生成** | 治本 | 镜像冗余类——schema 定义一次，生成前后端类型、校验、文档、迁移脚本 |
| **集中式注册表 + 派生** | 强 | 成对共变类——路由/事件/权限注册到一处，菜单/埋点/权限从注册表派生 |
| **契约测试 / 快照测试** | 强 | 断言「前后端 schema 一致」「文档与实现一致」 |
| **不变量测试** | 强 | 每个跨实体不变量写一条测试 |
| **显式标注 + 检查脚本** | 中 | 代码里标注「我关联着 B」，写脚本扫漏 |
| **关联登记表（文档）** | 弱，最后手段 | 上述都推不动的残余 |

举几个具体场景，看怎么「往左推」：

- **第 1 类镜像冗余** → 别再手写五份定义。用一处 schema（比如 Protobuf、Zod、Prisma、OpenAPI）作为唯一真相，用代码生成产出前后端类型和校验。关联从「隐性」直接跳到「不可能不一致」。
- **第 2 类成对共变** → 把路由、事件、权限集中注册到一个表/文件，导航菜单、埋点、权限校验都从这个注册表**派生**。加一个路由，菜单和权限自动跟上。
- **第 5 类跨实体不变量** → 写一条测试，断言「可用余额 + 冻结 = 总额」。不变量从此有守护。
- **第 11 类权限矩阵** → 权限判定逻辑只写一份，路由守卫、API 中间件、UI 显隐、数据过滤全部调用它。从「四处各写一遍」变成「一处定义四处引用」。

只有当一条关联实在无法用生成、注册表、测试来表达时（比如某些纯业务约定），才把它写进**关联登记表**。

## 八、文档的正确用法：关联登记表

回到最初的问题——「用文档强化关联」到底有没有用？有，但用法和大多数人想的不一样。

文档对关联的正确用法，**不是描述代码长什么样，而是登记那些推不动的隐性关联本身**。一张表就够：

| 当你改… | 必须同步改… | 为什么（意图） |
|---------|-----------|--------------|
| 订单状态枚举 | 三处 switch 分支 / 状态机文档 | 状态机驱动多分支行为 |
| User schema | 前端类型 / API 文档 / SDK | 同一概念的镜像投影 |
| 新增某路由 | 菜单 / 权限表 / 面包屑 | 成对登记约定 |
| 权限规则 | 路由守卫 / API / UI / 数据过滤 | 权限矩阵四处一致 |

这张表的本质，是给 AI（以及新人）的一份**强约束输入**：在改左列之前，先把右列喂进上下文，逐个处理。它把关联从「会被遗忘的背景知识」变成了「改动前的检查清单」。

但要注意，关联登记表本身也是文档，也会过时。所以它只能用于**那些无法用第七节手段推成显式的残余关联**，并且最好配合一个机制强制查阅——比如把「改 A 前先查登记表」固化成项目规则或自动化流程的固定步骤，而不是指望 AI 或人自觉去读。

## 九、收口：一套可落地的优先级

把全文压缩成一张行动清单：

1. **先判断关联属于 13 类里的哪一类。** 这是诊断。
2. **问自己：能不能用「生成 / 注册表 / 契约测试」把它往光谱左端推？** 能就推，这是治本。
3. **推不动的，进关联登记表。** 并且配机制强制查阅，而不是靠自觉。
4. **除了关联，优先记「禁忌 + 踩坑 + 被纠正的错误」**——这是代码完全表达不了、又防灾难的负向知识。
5. **永远不要记代码已经能回答的东西。** 那只会制造下一个过时的谎言。

最后强调一遍那个反直觉的判断，因为它是一切的起点：

> **关联丢失，不是 AI 不够聪明，而是关联没有被显式化。方向不是让 AI 记更多，而是让工具强制更多。**

当你发现一条关联反复被漏改，第一反应不该是「我得把它写进文档提醒 AI」，而应该是「我能不能让这条关联变得不可能被漏」——要么生成，要么测试，要么注册表。只有真做不到，才写进登记表。这一步思维转换，比任何 prompt 技巧都管用。
