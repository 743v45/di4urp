<!-- 注意：本文的编排原则是「主次优先」—— 越靠前的维度，对整体观感的影响越大。 -->

这篇文章列出评估一个网站「好不好看」时可以看的 500 个维度。

不是每次全过一遍。按照优先级排列——前面的维度决定了一个网站是 0 分还是 60 分，后面的维度决定它是 80 分还是 95 分。

---

## 第一优先级：骨架（决定及格与否）

这些是「做错一眼就能看出丑」的维度。一个网站如果这些不过关，后面的细节全是白费。

### 留白与呼吸感

留白是区分「业余」和「专业」最明显的标志。挤在一起的信息让人窒息，合理的留白让人愿意停留。

1. 元素之间是否有足够的留白
2. 留白是否按间距系统（4/8/16/24/32/48px）递进
3. 大区块之间的留白是否大于小区块内部
4. 留白分布是否均匀而非忽大忽小
5. 移动端的留白是否不因屏幕窄而压缩
6. 文字与容器边缘是否有足够的内边距
7. 图片与文字之间是否有呼吸空间
8. 卡片内部留白是否让内容不贴边
9. 留白是否传达了品牌调性（多=高端/少=信息密集）
10. 页脚是否与主体内容有明确间隔
11. 表单字段之间是否有足够间距避免误触
12. 首屏上方是否没有过多内容挤压
13. 是否有意识地用留白分隔功能区域
14. 留白方向是否一致（通常下方 > 上方）

### 对齐与基准线

没有对齐就没有秩序。对齐是设计的基础纪律。

15. 所有文字是否对齐到一致的基准线
16. 左对齐文本是否保持一致的起始位置
17. 居中元素是否真正居中（不是偏移）
18. 图标与文字是否垂直居中对齐
19. 表单标签与输入框是否对齐
20. 列表标记（圆点/数字）是否与内容对齐
21. 不同大小的元素是否对齐到栅格而非边缘
22. 头像+姓名+描述的组合是否对齐美观
23. 价格数字是否右对齐或小数点对齐
24. 对齐方式在同一区域内是否一致
25. 是否利用了光学对齐而非机械对齐（圆角元素视觉偏移补偿）
26. 底部对齐是否处理了不同高度元素的基准线
27. 多行文本与单行标签的混排是否对齐
28. 居中标题下的正文是否也居中
29. 移动端是否将部分居中布局改为左对齐

### 间距系统

间距的统一性是「精心设计」vs「随意摆放」的分水岭。

30. 是否使用了 4px 或 8px 为基数的间距系统
31. 所有间距是否来自预定义的 token（如 space-1/2/3/4）
32. 按钮内边距是否与字号成比例
33. 卡片内边距是否统一
34. 列表项间距是否一致
35. 区块间距是否随层级递增（组内 < 组间 < 区块间）
36. 水平间距与垂直间距是否有不同规则
37. 栅格间距（gutter）是否与间距系统协调
38. 边距是否在移动端适当缩小
39. 组件的 margin 是否只在一个方向设置（避免双向 margin）
40. 是否使用了 CSS 变量或 design token 管理间距
41. 响应式断点处间距是否做了特殊调整
42. 导航项之间的间距是否足够点击
43. 紧凑模式/宽松模式是否通过间距调整实现
44. 间距变化是否在过渡动画中平滑处理

### 颜色对比度与可读性

文字看不清，一切白搭。这是功能性底线。

45. 文字颜色对比度是否 >= 4.5:1（WCAG AA）
46. 大文字对比度是否 >= 3:1
47. 文字阴影/描边是否过度使用，显得廉价
48. 深色背景上的文字是否提高了对比度
49. 图片上的文字覆盖层是否有半透明遮罩保证可读
50. 占位符文字是否比输入文字略淡而非更小
51. 暗色模式的文字是否用浅灰而非纯白

### 字号层级

层级不清晰 = 读者不知道该先看哪里。

52. 是否建立了清晰的字号阶梯（如 12/14/16/20/24/32/48px）
53. 正文基准字号是否在 16–18px
54. H1 与正文的字号比是否 >= 2 倍
55. 各级标题之间是否有明确的级差
56. 辅助文字（时间戳/标签/脚注）是否不小于 12px
57. 按钮文字是否与按钮尺寸匹配
58. 大屏展示字号是否随视口等比放大
59. 是否使用 clamp() 做流式字号
60. 移动端正文是否适当放大到 17–19px
61. Hero 标题是否足够大以建立视觉冲击（通常 48px+）
62. 数据仪表盘中的数字字号是否突出
63. 输入框内文字是否与正文一致
64. 导航文字大小是否在 14–16px 之间
65. 脚注/法律条文的字号是否虽小但仍然可读

### 行宽与行高

66. 每行字符数是否控制在 45–75 个字符（中文 30–40 字）
67. 正文行高是否在 1.5–1.7 倍
68. 标题行高是否收紧到 1.1–1.3 倍
69. 短文本（按钮/标签）行高是否接近 1.0–1.2
70. 长文阅读行高是否放宽到 1.6–1.8
71. 行高是否随字号增大而收紧（反比关系）
72. 移动端行高是否比桌面端略宽
73. 多行标题的行高是否让人分清换行位置
74. 列表项之间的行高是否足够区分条目
75. 表格内的行高是否既紧凑又不拥挤
76. 代码块行高是否比正文略紧（1.3–1.5）
77. 极端字重下的行高是否做了单独调整
78. 段间距是否大于行高（用 margin 而非靠行高撑）

## 第二优先级：结构（决定舒适度）

骨架过关后，这些维度决定了用户在页面上是否「待得住」。

### 网格与栅格

79. 是否使用了统一的栅格系统（12 列/16 列/24 列）
80. 内容是否严格对齐到栅格
81. 栅格间距是否统一
82. 是否有合理的断点响应（sm/md/lg/xl/2xl）
83. 栅格在不同断点下是否合理合并列数
84. 是否利用了嵌套栅格处理复杂布局
85. 偏移（offset）使用是否有规律而非随意
86. 栅格外的元素（全宽背景）是否处理得当
87. 是否考虑了 RTL（从右到左）语言的栅格适配
88. 不规则布局是否有栅格逻辑支撑
89. 栅格是否同时在设计和开发中保持一致
90. 是否使用 auto-fill/auto-repeat 做自适应列数
91. 栅格对齐是否影响了文字可读性
92. 是否在必要时打破栅格创造视觉张力（有意为之）

### 容器与宽度

93. 正文容器是否限制在 640–760px
94. 整体容器是否限制在 1200–1440px
95. 全宽区域与内容宽区域的切换是否清晰
96. 侧边栏宽度是否合理（通常 240–320px）
97. 移动端是否去掉了不必要的侧边栏
98. 双栏布局的比例是否合理（如 2:1 或 3:1）
99. 三栏布局中主内容是否足够宽
100. 弹窗宽度是否与内容量匹配
101. 抽屉/侧面板宽度是否不超过屏幕的 80%
102. 卡片最大宽度是否限制（防止过宽难阅读）
103. 表格是否在窄屏时有水平滚动而非压缩
104. 图片容器是否有最大宽度防止过大
105. 内容区域是否居中展示
106. 左右边距是否随视口宽度自适应
107. 是否使用 container query 做组件级响应

### 布局逻辑

108. 视觉层次是否让用户视线自然从主到次流动
109. 内容优先级是否在第一眼就能看到
110. F 型 / Z 型阅读模式——关键内容是否沿自然阅读路径放置
111. 卡片布局一致性——同类卡片大小、间距、圆角是否统一
112. 列表与卡片的选择——信息密度高用列表，视觉展示用卡片，是否用对
113. 侧边栏是否喧宾夺主或过于狭窄
114. 全屏 vs 有边框——内容是撑满屏幕还是有限宽容器
115. 对称与不对称——是否有意识地选择来表达品牌
116. 内容密度是否匹配定位（密集信息型 vs 宽松品牌型）
117. 折叠线以上（above the fold）——首屏是否传达了核心价值主张
118. 垂直节奏——各区块的高度是否有韵律感，不忽大忽小
119. 水平对齐——左对齐/居中/右对齐是否一致不混乱
120. 模块化结构——页面是否由清晰可辨的区块组成
121. Hero 区域是否吸引人且传达信息
122. CTA 按钮是否在视觉流末端，容易点击
123. 信息架构——导航层级是否扁平，3 次点击内到达目标
124. 页面分区标识——不同内容区块是否有清晰的视觉分隔
125. 骨架屏/加载占位——布局加载时是否保持结构稳定

### 颜色体系

颜色是情感的第一入口。用错颜色，后面的细节没人看。

126. 是否有明确的品牌主色
127. 是否有不超过 1–2 个辅助色
128. 中性色是否有完整的灰度阶梯（至少 5 级）
129. 是否定义了语义色（成功/警告/错误/信息）
130. 配色方案是否避免了彩虹式混搭
131. 颜色是否通过 HSL 系统生成而非随意拾取
132. 暗色模式是否重新设计了配色（不是简单反色）
133. 暗色模式的背景是否用深灰而非纯黑
134. 高饱和度颜色是否只用于小面积强调
135. 大面积背景是否使用了低饱和度的颜色
136. 同一色调是否在不同元素上一致使用
137. 颜色命名是否基于用途（primary/danger）而非色值（blue/red）
138. 渐变色是否有明确的起止点，方向是否统一
139. 透明度使用是否克制且有规律（如统一用 4/8/12/16 的 alpha）
140. 不同色盲类型下关键信息是否仍可区分
141. 品牌色在页面上的出现频率是否足够建立印象
142. 空白区域的「白色」是否与整体色调协调
143. 焦点色是否与品牌色有关联
144. 禁用状态是否用低饱和度/低透明度表达
145. 悬停色是否是原色的加深或提亮变体
146. 是否避免了红绿对比（色盲不友好）
147. 颜色与情感传达是否匹配（蓝色=信任/绿色=健康/橙色=活力）

### 圆角与边框

圆角和边框是「统一感」的重要来源。混用是大忌。

148. 全站圆角大小是否统一（如 4/8/12/16px）
149. 大元素是否用更大的圆角
150. 小元素（徽章/标签）是否用全圆角或小圆角
151. 按钮、输入框、卡片的圆角是否协调
152. 圆角大小是否与品牌调性匹配（大=友好/小=严谨）
153. 同一元素内外层圆角是否协调（嵌套时内层圆角更小）
154. 边框颜色是否使用低对比度而非纯黑
155. 边框粗细是否统一（通常 1px）
156. 是否有意识地选择有边框 vs 无边框+阴影
157. 分割线是否比边框更轻（更淡的颜色或虚线）
158. 表格边框是否只保留必要的（通常只要底边）
159. 卡片是否有边框/阴影/或纯靠留白区分
160. 输入框边框在 focus 时是否有变化
161. 圆角是否在悬停/激活状态保持一致
162. 全圆角（pill shape）是否只用于特定场景（标签/特殊按钮）

## 第三优先级：排版（决定品质感）

骨架和结构过关后，排版是拉开差距的关键。好的排版让页面「有教养」。

### 字体家族

163. 主字体是否匹配品牌气质（科技/人文/奢饰/童趣）
164. 是否控制在 2–3 种字体以内
165. 中文字体是否做了针对性选择（思源/霞鹜/站酷等）
166. 英文字体是否与中文字体风格协调
167. 标题字体是否有足够的辨识度和个性
168. 正文字体是否选用了高可读性的经典字体
169. 是否避免了过度使用系统默认字体（除非有意为之）
170. 装饰性字体是否只在极少数位置使用
171. 等宽字体是否在代码/数据场景正确使用
172. 字体是否支持目标语言的所有字符集
173. 是否考虑了字体许可证和版权问题
174. Web 字体是否做了子集化（subset）减少加载量
175. 是否有 fallback 字体链，保证降级时风格接近
176. 品牌定制字体是否在关键位置突出使用
177. 不同操作系统下字体渲染是否都可接受

### 字重与样式

178. 是否使用了至少 2–3 种字重来建立层级
179. 正文是否用 Regular（400）而非 Light（300）
180. 标题是否用 Bold/Semibold 而非单纯放大字号
181. 是否避免了用 italic 做强调（中文排版中 italic 基本无效）
182. 粗体是否只用于真正需要强调的地方
183. 字重变化是否与信息层级一一对应
184. 链接文字是否通过颜色而非字重来区分
185. 标签/徽章是否用了适当的字重
186. 过轻的字重（100/200）是否在小字号下仍清晰
187. 数字展示是否用了较重的字重增强存在感
188. 正文与引用是否通过字重做了区分
189. 按钮文字是否用了中高字重（500–600）
190. 导航项 active 状态是否通过字重变化标识
191. 是否避免了全篇用同一字重的单调感
192. 字重过渡是否自然（不跳级，如从 400 直接到 700）

### 字间距与段落

193. 大标题是否适当收紧 letter-spacing（-0.02em）
194. 全大写文字是否加宽 letter-spacing（0.05–0.1em）
195. 中文正文是否不需要额外 letter-spacing
196. 行高是否随字号增大而收紧
197. 段间距是否为正文字号的 1–1.5 倍
198. 段落首行是否用了缩进（中文 2em）或段间距来分隔（二选一）
199. 是否避免了过长的行宽导致阅读疲劳
200. 是否避免了过窄的行宽导致频繁换行
201. 两端对齐是否造成了过多的 river（白色河流）
202. 中文排版是否优先使用左对齐
203. 文字是否避开了底色复杂的图片区域
204. 长文是否有子标题打断，不让读者面对文字墙
205. 引用块是否有明显的样式区分
206. 列表排版是否对齐美观
207. 链接在段落中是否通过下划线或颜色清晰标识
208. 标点符号是否做了悬挂处理（避免行首出现标点）
209. 中英文之间是否加了自动空格
210. 数字与中文之间是否有合适的间距

## 第四优先级：组件（决定统一感）

组件的一致性是「有体系」vs「东拼西凑」的分界线。

### 按钮与交互元素

211. 按钮是否有明确的层级（主要/次要/文字/危险）
212. 主按钮颜色是否足够突出
213. 次要按钮是否不抢主按钮的视觉权重
214. 按钮内边距是否足够点击（至少 44x44px 触控区）
215. 按钮文字是否简洁（2–4 字）
216. 按钮圆角是否与其他元素协调
217. 按钮 hover 状态是否有明确反馈
218. 按钮 active/pressed 状态是否有缩放/阴影变化
219. 禁用状态是否视觉上明显区分
220. 按钮组之间的间距是否合理
221. 图标按钮是否提供了 tooltip
222. 按钮是否使用了合适的 HTML 元素（button 而非 div）
223. 满宽按钮是否只在移动端使用
224. 按钮内文字是否不换行
225. Loading 状态的按钮是否有 spinner 替代文字
226. 切换按钮/开关的 on/off 状态是否视觉明确
227. 复选框/单选框的自定义样式是否统一
228. 下拉菜单的触发按钮是否有指示箭头
229. 按钮是否避免了过多颜色变体
230. 浮动操作按钮（FAB）位置是否不遮挡内容

### 表单设计

231. 输入框高度是否与按钮一致
232. 标签位置是否统一（上方或左侧，不混用）
233. 占位符是否用 lighter 颜色而非更小字号
234. 错误提示是否在字段旁边即时显示
235. 必填标记是否统一（星号或文字说明）
236. 输入框 focus 状态是否有清晰的边框/阴影变化
237. 表单验证是否用颜色+图标+文字三重反馈
238. 密码输入是否有显示/隐藏切换
239. 长表单是否分步/分组
240. 搜索框是否有搜索图标和清除按钮
241. 下拉选择是否样式统一
242. 文件上传区域是否有拖拽提示
243. 表单提交后的成功/失败反馈是否明显
244. 自动补全的样式是否与输入框协调
245. 表单字段顺序是否符合用户预期
246. 多行文本框是否可调整大小
247. 日期选择器样式是否与系统统一
248. 表单整体的视觉重量是否与页面其他部分平衡
249. 输入框内是否有足够的 padding
250. disabled 字段的样式是否明显区分

## 使用方法

**快速筛查（5 分钟）**：只看第一优先级（1–92 条）。这 92 条决定了网站是否及格。

**标准审查（15 分钟）**：过前四个优先级（1–320 条）。覆盖了骨架、结构、排版、组件。

**完整审查（30 分钟+）**：全部 500 条。适合上线前的最终检查。

**工具验证**：配合 Lighthouse、WAVE、Chrome DevTools 的色盲模拟和 Responsively 等工具，自动化一部分检查。
