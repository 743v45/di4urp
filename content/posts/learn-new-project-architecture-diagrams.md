## 为什么需要这张路线图

接手一个新项目时，最常见的困境是**不知道从哪里看起**。代码库庞大、模块众多、文档零散——随便翻开一个文件，很快就会迷失在细节里。

根本原因是：**你缺少一张地图**。

架构图就是这张地图。但"画架构图"本身也是个模糊的指令——画什么图？画到什么程度？先画哪张？有没有优先级？

这篇文章把"从零学习一个新项目"拆解成 **13 个认知阶段、36 类架构图**，按从宏观到微观的顺序排列。每类图标注了核心程度和是否必画，让你能按图索骥，高效建立对陌生项目的全局理解。

---

## 与 C4 模型的关系

[C4 模型](https://c4model.com/) 是业界广泛使用的架构可视化标准，将系统分为四个抽象层次：Context（系统上下文）、Container（容器/服务）、Component（组件）、Code（代码级）。

本文的认知路线图与 C4 的对应关系：

| C4 层次 | 对应认知阶段 |
|---------|------------|
| Context | 阶段 1：建立全局感 |
| Container | 阶段 3-4：理解数据流 + 理解运行时 |
| Component | 阶段 5-6：理解控制流 + 理解部署运维 |
| Code | 阶段 9：理解开发体验 |

本文在 C4 的基础上做了两件事：**前置了业务语义理解**（C4 不涉及业务概念），并**扩展了非功能性维度**（性能、安全、故障模式等）。建议画图时参考 C4 的图标规范和分层思路。

## 核心原则

在展开表格之前，有三条原则需要先说清楚：

**原则一：先全局后细节。** 不要一上来就钻进某个模块的内部逻辑。先搞清楚系统边界、业务概念、模块结构，再深入具体实现。这就是为什么表格把"建立全局感"排在第一个阶段。

**原则二：跟着数据走。** 理解一个系统最有效的方式不是理解每个模块的代码，而是理解**数据怎么流过这些模块**。追踪一个请求从入口到响应的完整路径，比读十个文件的源码更有价值。

**原则三：够用就停。** 不是每个项目都需要画满所有图。必画的 6 张搞定就能改代码了，其余按需补充。画图是为了理解，不是为了完成任务。

## 必画清单速查

上面 36 类图里，标记了 **✅ 必画** 的共 6 张：

| # | 图类型 | 推荐图类型 | 画什么 |
|---|--------|-----------|--------|
| 1 | 系统上下文图 | C4 Context / 框图 | 系统边界、外部依赖 |
| 2 | 模块依赖图 | 有向依赖图（DAG） | 代码顶层组织结构 |
| 8 | 核心数据流图 | DFD 数据流图 | 主请求的端到端路径 |
| 9 | 数据模型 / ER 图 | ER 图 | 核心实体和关系 |
| 12 | 进程 / 服务拓扑图 | 拓扑图 / C4 Container | 运行时进程结构 |
| 13 | 时序图 | UML 时序图 | 核心场景的交互顺序 |

### 为什么是这 6 张

选择标准：**覆盖系统静态结构 + 动态行为的最小集合**。

- 系统上下文图 + 模块依赖图 → 解决"系统长什么样"（静态结构）
- 核心数据流图 + 时序图 → 解决"运行时怎么跑"（动态行为）
- 数据模型 / ER 图 → 解决"核心概念是什么"（领域认知）
- 进程 / 服务拓扑图 → 解决"部署起来是什么样"（运行时结构）

这 6 张图构成一个闭环：**结构 → 行为 → 数据 → 部署**。搞定后你就能定位问题、改代码、做 code review。

不同项目可以根据业务特点调整。例如，工作流密集的系统可以把状态机图加入必画；微服务架构可以把 API 接口地图加入必画。

## 实操建议

### 一、不要一次性全画

很多人拿到这套框架后第一反应是"把所有图全画完"。这是最常见的时间浪费。正确的做法是**按需画、够用就停**：

- **接手新项目前两周：** 画阶段 1-4 的必画 6 张
- **需要改代码时：** 补充阶段 5（理解控制流）的控制流图
- **需要排故障时：** 补充阶段 6（理解部署运维）的运维图 + 阶段 13（理解故障模式）的故障图
- **需要做架构决策时：** 补充阶段 8（理解演进能力）+ 阶段 10（理解性能特征）

### 二、图是给自己画的，不是给汇报用的

架构图的价值在于**画的过程中学到的**，而不是画完后的产出。所以：

- 用最简单的工具：白板、纸笔、Mermaid、Excalidraw
- 不追求美观，追求准确
- 边画边验证：对照代码确认你的理解是否正确
- 理解加深后随时更新，不要怕推翻重画

### 三、选一条主线追踪

学项目最怕"到处看但哪都没看透"。建议选**一条核心业务主线**（比如"用户下单"或"Issue 从创建到关闭"），从阶段 1 到阶段 5 都用这条主线贯穿。一条线追踪到底，比十条线各看一段有效得多。

### 四、每个阶段都有"过线标准"

不是画完图就算理解了。每个阶段都有检验标准：

| 阶段 | 检验问题 |
|------|---------|
| 阶段 1：建立全局感 | 你能在一分钟内向新人描述这个系统吗？ |
| 阶段 2：理解业务语义 | 你能解释项目里核心业务术语的含义吗？ |
| 阶段 3：理解数据流 | 你能画出主业务请求的完整路径吗？ |
| 阶段 4：理解运行时 | 你能说出系统跑起来有几个进程吗？ |
| 阶段 5：理解控制流 | 你知道一个请求被拒绝是怎么决定的吗？ |
| 阶段 6：理解部署运维 | 你知道代码怎么部署上线的吗？ |
| 阶段 7-13 | 按需，能回答该阶段的核心问题即可 |

如果答不上来，说明这阶段还没过，需要回去补。

### 五、结合代码阅读

图是骨架，代码是血肉。画图和读代码要交替进行：

1. 先画一张粗略的图（基于 README、目录结构、文档）
2. 对照代码验证和补充
3. 发现理解有误就修正图
4. 重复直到图和代码一致

这个过程本身就是最高效的学习方式。

## 常见误区

**误区一："先把代码全读完再画图"**

读完所有代码再画图，等于背完整本地图册再出门。正确的顺序是：先画粗图 → 对照代码修正 → 逐步细化。

**误区二："架构图只在入职时画一次"**

架构图是**活的文档**。代码在演进，图也应该跟着更新。每次做架构变更后花 10 分钟更新相关图，比每隔半年重画一次高效得多。

**误区三："每张图都要画到 production 质量"**

大部分图只需要自己能看懂就够了。只有需要分享给团队、写进设计文档的图才需要打磨格式。内部学习用的图，手绘草图就足够。

**误区四："图越多越好"**

36 类图不是检查清单，是工具箱。你不会同时用到工具箱里的每一件工具。解决当前问题的那张图画好，比所有图都画但都画得马虎有价值得多。

## 写在最后

从零学习一个新项目，本质上是一个**建立心智模型**的过程。架构图是这个过程中最高效的工具——它强迫你把模糊的理解变成明确的表达，把碎片的信息组织成结构的知识。

13 个阶段，6 张必画，按认知排序，够用就停。这就是全部的方法论。

剩下的，就是选一个项目，选一条主线，开始画第一张图。
